# 01_封面

各位老师、同学好，我们是第十二组。今天要汇报的题目是『基于机器视觉的农产品品质分级与缺陷检测系统』。我们把它落到一个非常具体的方向上——面向肇庆本地的柑橘，做一个能够给出等级、框出表面缺陷、并提供量化依据的 Web 系统，我们给它取名叫 CitrusVision。接下来由我代表小组，用大约十分钟，把我们的整体方案讲清楚。

---

# 02_汇报路线图

在正式展开之前，先用一页交代今天的讲解路线。我们会按六步推进：先讲为什么做，包括项目背景和选题论证；再讲做什么，也就是定位、用户、需求和功能；然后是怎么做，涵盖技术、架构、算法、流程和数据库；接着是怎么推进，包括实施计划和团队分工；之后是如何验证，也就是测试和验收；最后落到预期结果，包括创新、风险和交付物。这条主线刻意对齐了软件工程的完整过程，方便老师对照评分。

---

# 03_项目背景与问题痛点

我们的出发点是一个真实的痛点。肇庆的四会、德庆是重要的柑橘产区，但目前的品质分级主要还靠人工目检，主观性强、标准不统一、效率也低，表面的裂纹和病斑容易漏检，更谈不上溯源。机器视觉恰好能补上这一块——把分级做得客观、可量化、还可追溯。所以我们把要解决的问题定义得很清楚：让一张照片，在几秒钟内就得到等级、缺陷和量化依据。

---

# 04_选题论证

那为什么偏偏选柑橘？我们在五个候选品类里做了一次横向比较，从地方特色、缺陷维度、形态分割难度、分级直观性、数据可得性这五个维度逐一打分。柑橘的综合表现最好：它紧扣题目给的第一个示例，又有肇庆的地方特色，缺陷类型清晰、果形规整便于分割、分级标准也直观。相比之下，叶菜形态太复杂、香蕉的缺陷维度偏单一，所以我们最终锁定柑橘。

---

# 05_项目定位与核心目标

选题确定之后，再来说定位。我们把系统定位成一个『教学演示加产地初分』的、工程完整的四层 Web 应用——不追求工业级精度，但要求整条工程链路是完整的。围绕这个定位，我们设了几个可量化的目标：单张检测三秒以内返回，缺陷检测的平均精度均值不低于零点六，分级结果与人工的一致率达到八成以上，支持一次至少十张的批量处理，结果能生成带二维码的报告，并且全套 UML 文档齐备。这每一条，都是可以在答辩现场逐项核验的。

---

# 06_目标用户与使用场景

这些目标最终都要落到具体的人身上。我们的系统服务三类角色：普通用户，也就是分拣员和合作社质检员，他们的主场景是微信扫码、拍照分级；管理员负责维护分级标准、价格和模型；还有答辩现场的评委，属于间接用户。最典型的使用旅程是这样的——质检员扫码进入手机页面，拍照或批量上传，秒级看到等级和缺陷红框，按等级做统计并关联参考价，最后一键导出当批报告。

---

# 07_需求分析

基于这些场景，我们做了完整的需求分析。左边是用例图，两类参与者，普通用户和管理员，围绕上传、检测、分级、溯源、导出和后台管理这几个核心用例展开。功能性需求我们一共梳理了十四项并做了分组；非功能性需求也定得很具体，比如单张三秒、批量三十秒以内、至少支持五个并发，再加上可用性和兼容性。同时我们没有忘记课程设计本身的交付要求——UML 文档、代码注释、还有不超过五十兆的演示视频。

---

# 08_功能模块设计与优先级

功能虽然多，但不能一把抓，所以我们用同心圈做了优先级划分。最内核是必做的 P0，是必须先跑通的闭环：登录、上传、预处理、缺陷检测、分级、可视化、入库和统计。中间一圈 P1 是加分项，比如溯源二维码、价格、报告导出和样本日志。最外圈 P2 是有余力再做的扩展，像模型切换、人工复核、多品类和缺陷热力图。我们的取舍原则很简单——先保闭环，再做加分。

---

# 09_核心业务流程

接下来看核心业务流程是怎么串起来的。用户登录后上传图片，后端先做校验、存图、建立检测任务，然后调用独立的算法服务做推理；算法服务内部走预处理、果实分割、缺陷检测、特征计算、规则分级这一串，再把结果回写、关联价格和溯源，最后由前端渲染、支持导出。这里有一个关键设计——后端只做编排、不做推理，算法独立成一个服务，这样职责清晰、解耦，也方便我们四个人并行开发。

---

# 10_技术栈选型

技术栈的每一项，我们都有明确的理由。前端用 Vue3 加 ECharts，出活快、图表好，一套响应式同时覆盖电脑和手机；后端用 FastAPI，和算法同语言便于集成，还自带接口文档；算法服务用 OpenCV 加 YOLOv8，独立部署、用 CPU 也能推理；数据库用 MySQL 8，便于画 E-R 图、表达关系；训练则放在 Colab 或者 AI Studio 的免费 GPU 上，正好规避了我们本地没有 GPU 的现实问题。

---

# 11_系统总体架构

把这些技术组织起来，就是这张四层架构图。从上到下分别是前端层、后端 API 层、算法服务层和数据层：前端覆盖电脑端和微信手机页；后端用 FastAPI 做登录鉴权和业务编排；算法层是独立进程，对外只暴露一个推理接口；数据层包含数据库、文件存储和模型权重。各层之间用 HTTP 和 JSON 通信。这样分层的依据是职责单一、四个人能够并行，而且每一层都能对应到我们的包图。

---

# 12_核心算法方案

算法是这个项目的难点，我们的方案是双引擎、可解释、可兜底。一条引擎是传统计算机视觉，负责量化分级；另一条是 YOLOv8，负责缺陷检测，两者融合。整个管线分五步：预处理、果实分割、缺陷检测、特征计算、规则分级。我们会算出三个量化指标——果径、着色率、缺陷占比，再用规则评定一级、二级还是等外品。万一深度学习模型表现不稳定，传统视觉可以兜底，保证系统始终是可用的。

---

# 13_数据库设计

算法和业务产生的数据，落到数据库就是这张 E-R 图。我们用 MySQL 8 设计了十张核心表，主线是『一次检测任务对应多条结果，每条结果又对应多个缺陷框』。围绕这条主线，用户、样本、任务、结果、缺陷是核心实体；而分级标准、价格、溯源、模型、日志，都拆成了独立可维护的表。这样的好处是，管理员能在线调整阈值而不用动代码。所有表的主键统一用 BIGINT，并且都带创建和更新时间。

---

# 14_项目实施计划与里程碑

有了方案，再看我们怎么按软件工程的节奏推进。我们按四周来规划，每一周对应一个阶段，阶段的产出就是里程碑：第一周立项与需求，产出 M1；第二周设计与原型，产出 M2；第三周编码实现，跑通 MVP 闭环，是 M3；第四周联调、测试、交付和答辩排练，是 M4。这里我要特别强调关键路径——数据采集、模型训练、系统联调这条链最长，所以数据采集我们第一周就并行启动，绝不会等到训练前才去找数据。如果实际课程周数不同，我们准备了三周的压缩版，但 UML 文档不会删减。

---

# 15_团队分工与协作机制

好的计划要靠合理的分工来落地。我是组长[组长]，负责总体架构、后端、数据库和集成联调，同时统稿报告、组织答辩；[成员A]是算法负责人，主攻 YOLOv8 的训练调优和部署；[成员B]是算法副手兼数据工程，负责数据集构建、传统视觉分级和需求分析；[成员C]负责前端、测试和文档。我们的考量是——算法是难点，所以配了双人攻坚；后端是集成枢纽，由我来把控；前端、测试、文档由一个人贯通，保证演示和交付。协作上我们接口契约先行，用 Git 分支并行开发，再由我合并，文档分阶段统稿。

---

# 16_测试方案与验收标准

质量怎么保证？我们设计了五层测试：功能测试用用例表，接口测试用 Postman 加 pytest 自动验证，算法评估用 mAP 和混淆矩阵，系统联调做端到端和并发，再加上多浏览器和手机的兼容测试。我们还准备了一批典型用例，比如上传非图片要被拒绝、裂纹果要判等外品并框出、批量十张要全部返回且统计正确、未登录调接口要返回四零一。在这个基础上，我们给系统下了明确的『完成定义』：基础要求是 MVP 闭环没有致命 bug，算法要求 mAP 不低于零点六、一致率八成以上，加分功能可演示、文档齐全、按时按规范提交。每一条都能逐项核验。

---

# 17_创新亮点与风险应对

再说我们的差异化亮点和风险预案。创新上主推三点：第一，双引擎可解释分级，分级是有量化依据的、不是黑盒；第二，分级标准可配置、模板化，内置了砂糖橘和贡柑的模板，管理员能在线调阈值；第三，溯源二维码和参考价格联动，结果能一键生成可扫码的溯源报告。第四点多品类扩展，作为有余力时的扩展项。风险方面我们也想在前面：数据不足，就用公开数据集打底、Roboflow 加速标注；模型不稳，就用传统视觉兜底；没有 GPU，就上免费云 GPU 和轻量模型；现场出故障，就用预录视频做双保险。我们把每一个风险都提前降到了中低等级。
