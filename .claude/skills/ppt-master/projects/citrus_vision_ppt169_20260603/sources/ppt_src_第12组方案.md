# 二、推荐方向

## 2.1 推荐结论（一句话）
> **在题目10框架内，聚焦"肇庆本地柑橘（以砂糖橘/贡柑为代表）"单一品类，构建"传统视觉做可量化品质分级 + 轻量深度学习（YOLOv8）做表面缺陷检测"的双引擎 Web 系统，数据采用"公开数据集 + 网络采集筛选 + 自建小样本"三源混合，演示级规模即可。**

即：检测对象收敛到"柑橘"，算法走"传统 CV 分级 + DL 缺陷检测"混合路线，系统做成完整的前后端 + 独立算法服务 + 数据库四层 Web 应用。

## 2.2 为什么是"柑橘"

| 候选对象 | 地方特色 | 缺陷维度 | 形态/分割 | 分级直观 | 数据可得 | 评价 |
|---|---|---|---|---|---|---|
| **柑橘（砂糖橘/贡柑）** | ★★★ | ★★★ | ★★★ | ★★★ | ★★ | **✅ 首选** |
| 苹果 | ★ | ★★★ | ★★★ | ★★★ | ★★★ | 备选数据源 |
| 香蕉 | ★★ | ★ | ★★ | ★★★ | ★★ | 缺陷功能撑不起 |
| 番茄 | ★ | ★★ | ★★ | ★★ | ★★★ | 扩展验证对象 |
| 叶菜蔬菜 | ★★ | ★ | ★ | ★ | ★ | ❌ 风险高 |

**关键理由**：题目把"柑橘"列示例第一位；肇庆是著名柑橘产区（四会柑/德庆贡柑/砂糖橘），地方特色是答辩天然加分；近圆形、表皮光滑、背景易控，目标检测与色泽提取都简单，但缺陷类型丰富，"分级+缺陷检测"两功能都站得住；着色率=成熟度，肉眼可验证，可解释性强。数据是唯一短板，由 §11 三源混合解决，且课程只需演示级规模。

# 三、详细项目脚手架方案

## 1. 项目名称
- **正式中文名**：**基于机器视觉的肇庆柑橘品质分级与缺陷检测系统**
- **工程代号**：**CitrusVision — Citrus Quality Grading & Defect Detection System**
- **提交压缩包命名**（红线）：`第12组-《基于机器视觉的农产品品质分级与缺陷检测系统》`
- 说明：报告/答辩用全称（不偏离原题名），系统品牌名 "CitrusVision"，仓库/目录名 `citrus-vision`。

## 2. 项目背景与问题定义
- **现实意义**：肇庆（四会、德庆等）是广东重要柑橘产区。传统品质分级靠人工目检，存在主观性强、标准不一、效率低、缺陷漏检、可追溯性差等问题。
- **问题边界**：
  - **做**：单张/批量柑橘图像的品质分级（一级/二级/等外品）+ 表面缺陷检测与框选（病斑/裂纹/碰伤/畸形）+ 溯源信息登记 + 市场价格参考 + 结果可视化与报表导出。
  - **不做**：产线实时高速分拣硬件；多品类通用模型；内部品质（糖度、内部腐烂）无损检测；真实传感器/PLC 接入。
- **为何选柑橘**：见 §2.2，核心是"地方特色+缺陷丰富+形态易检测+演示直观+风险可控"。

## 3. 项目定位与核心目标
- **定位**：面向教学演示与产地初分场景的、工程完整的机器视觉品质检测 Web 系统（B/S，四层分离）。
- **核心目标（可量化、可验收）**：
  1. 上传柑橘图像，**≤3 秒**（单张 CPU）返回：等级标签 + 缺陷红框 + 量化指标（果径估计、着色率%、缺陷面积占比%）+ 置信度。
  2. 缺陷检测 **mAP@0.5 ≥ 0.60**（演示级目标，非硬指标）。
  3. 分级与人工标注**一致率 ≥ 80%**（演示集）。
  4. 支持**批量上传 ≥ 10 张**并汇总统计。
  5. 每条结果可生成含**溯源信息+二维码**的 PDF/Excel 报告。
  6. 形成**完整软件工程文档**（用例图、DFD 1 层、数据字典、软件结构图、类图、时序图、活动图、状态图、E-R 图齐备）。
- **输出形态**：结构化结果(JSON) + 可视化标注图 + 分级徽章 + 统计图表 + 可导出报告。

## 4. 目标用户与应用场景

| 用户角色 | 场景 | 典型流程 |
|---|---|---|
| 产地分拣员/合作社质检员（普通用户） | 收果初分、电商上架前分级 | 登录→拍照/上传→看等级与缺陷→登记溯源→导出报告 |
| 管理员/技术员 | 维护标准、管理数据与模型 | 登录→维护分级标准/价格→管理样本库→切换模型→看日志统计 |
| 答辩评委/演示观众（间接） | 现场演示 | 观看实时检测、批量处理、报告导出 |

- **主场景**：质检员手机微信扫码进 H5，对一筐砂糖橘逐个/批量拍照上传，系统秒级返回等级与缺陷标注，按等级计数并给参考价格，导出当批分级报告。

## 5. 推荐的细分检测对象（落地结论）
- **主对象：柑橘 — 以"砂糖橘/贡柑"为代表**（单一品类，全周期聚焦）。
- **缺陷类别（建议 4 类，可精简为 3 类）**：`病斑(black_spot)`、`裂纹(crack)`、`碰伤(bruise)`、`畸形(deformity)`；周期紧时合并为 `病斑/裂纹/损伤`。
- **品质维度**：尺寸（果径）、色泽（着色率/成熟度）、表面缺陷（面积占比+类型）。
- **扩展对象（仅加分/鲁棒性验证）**：用公开苹果/番茄数据集做对照实验，证明系统可扩展到多品类。

## 6. 需求分析
**6.1 功能需求（FR）**
- FR1 注册/登录/登出，角色区分（管理员/普通用户）。
- FR2 图像上传：PC 拖拽 + 移动端 H5 拍照 + 单张与批量（≥10 张）。
- FR3 预处理：尺寸归一化、去噪、光照/白平衡校正、ROI 提取（果实分割/背景剔除）。
- FR4 品质分级：计算果径、着色率、缺陷占比，按预设标准评一级/二级/等外品。
- FR5 缺陷检测：定位并框选病斑/裂纹/碰伤/畸形，输出类别与置信度。
- FR6 结果可视化：标注图、量化指标卡片、等级徽章、批量结果列表与统计图。
- FR7 溯源信息：录入/展示产地、品种、批次、检测时间、检测人，生成溯源二维码。
- FR8 价格参考：按等级映射市场参考价（管理员可维护）。
- FR9 报表导出：单条/批量导出 PDF、Excel。
- FR10 样本管理：样本入库、检索、删除；标注数据管理。
- FR11 模型管理（管理员）：版本登记、激活/切换、查看评估指标。
- FR12 标准管理（管理员）：维护分级阈值。
- FR13 日志管理：记录关键操作与检测记录，支持查询。
- FR14 统计看板：检测总量、各等级占比、缺陷类型分布。

**6.2 性能需求**：单张 ≤3s(CPU)/≤1s(GPU)；批量 10 张 ≤30s；演示级 ≥5 并发不崩；jpg/png，单张 ≤10MB。
**6.3 可用性**：界面中文、检测 ≤3 步、上传带进度、错误友好提示、H5 适配主流手机。
**6.4 展示需求**：缺陷框选醒目、分级一目了然、批量有图表、支持实拍演示与一键导出。
**6.5 课设交付需求**：需求/概要/详细 UML 齐全且详细；代码注释；视频 ≤50M；按命名规范打包；按时提交。

## 10. 算法方案设计
**总路线：混合方案（传统 CV 做分级 + 深度学习做缺陷检测），双引擎、可解释、可兜底。**

**10.1 处理流程**
```
原图 → ①预处理 → ②果实分割/ROI → ③缺陷检测(YOLOv8) → ④特征计算 → ⑤分级评定 → 结果
```
1. 预处理：缩放归一化、高斯去噪、灰度世界白平衡、（可选）CLAHE 增强对比。
2. 果实分割/ROI：HSV 阈值 + 形态学开闭运算分离果实与背景，得果实掩膜与外接圆（用于果径估计）。
3. 缺陷检测（DL）：YOLOv8n/s 检测病斑/裂纹/碰伤/畸形，输出框+类别+置信度。
4. 特征计算（传统 CV）：
   - 果径：掩膜外接圆/最小外接矩形像素直径 + 参照物或固定拍距换算 mm（演示可用相对尺寸）。
   - 着色率：果实区域内橙色像素(HSV)占比，表征成熟度/色泽。
   - 缺陷占比：缺陷框/掩膜总面积 ÷ 果面面积。
5. 分级评定（规则引擎，可解释）— 自拟标准（A5）：

| 等级 | 判定规则（满足全部） |
|---|---|
| 一级（精品） | 果径达标 且 着色率 ≥ 85% 且 缺陷占比 < 2% 且 无裂纹/腐烂 |
| 二级（合格） | 着色率 ≥ 70% 且 缺陷占比 < 5% 且 无裂纹/严重畸形 |
| 等外品 | 着色率 < 70% 或 缺陷占比 ≥ 5% 或 存在裂纹/严重畸形/腐烂 |

阈值由管理员在"标准管理"模块可配置，报告中说明依据。

**10.2 数据需求/标注/模型/评估**
- 数据需求：缺陷检测每类 ≥80~150 张；分级每等级 ≥50 张；演示级总量 300~600 张。
- 标注：缺陷用矩形框（YOLO 格式，LabelImg/Roboflow）；分级用图像级类别标签（文件夹分类）；制定《标注规范》。
- 模型选择：YOLOv8n（CPU 友好）优先，不足升 YOLOv8s；分级以规则引擎为主，**可选**加迁移学习 MobileNetV2/ResNet18 分类器对照。依据：小样本+无强 GPU+要可解释+要兜底。
- 评估：缺陷检测 mAP@0.5/Precision/Recall/各类 AP；分级准确率/混淆矩阵/人工一致率；报告附 PR 曲线、混淆矩阵。
- 兜底：DL 欠佳时传统 CV 分级链路独立可用，保证演示有结果。

## 11. 数据集方案
- **三源混合**：① 公开数据集打底（Kaggle/百度 AI Studio/天池 的柑橘/柑橘类/水果质检缺陷数据集；扩展对照用苹果/番茄）；② 网络采集筛选（搜索引擎+电商商品图，人工剔除水印/重复/无关）；③ 自建小样本（组员手机拍不同等级/缺陷柑橘，纯色背景、固定距离、放硬币参照，每类几十张）。
- **数据组织**：
```
data/
  raw/                 # 原始采集
  images/{train,val,test}/
  labels/{train,val,test}/   # YOLO txt
  grade/{grade1,grade2,grade_out}/  # 分级分类
  annotations/         # 标注源文件
```
- **标注规则**：统一缺陷类别定义与最小框尺寸；模糊/重叠缺陷处理约定写入《标注规范》。
- **划分**：训练:验证:测试 = **7:2:1**，按等级/缺陷类别分层抽样。
- **增强**：旋转、翻转、亮度/对比度抖动、加噪、Mosaic（YOLOv8 内置），扩 3~5 倍。
- **周期内最可行**：W1~W2 并行以"公开+网络采集"快速凑齐缺陷训练集；自建小样本主要做演示和测试集（演示用肇庆本地实拍强化地方特色）。

## 12. 系统总体架构
**四层 B/S（前端 / 后端 API / 算法服务 / 数据库）+ 文件存储：**
```
┌──────────────┐     HTTP/JSON      ┌──────────────────┐
│  前端 (Vue3)  │  ───────────────▶  │  后端 API (FastAPI)│
│ PC Web + H5  │  ◀───────────────  │  鉴权/业务/编排    │
└──────────────┘                    └────────┬─────────┘
                                              │ 内部 HTTP 调用 /infer
                                     ┌────────▼─────────┐
                                     │ 算法服务 (FastAPI) │
                                     │ OpenCV+YOLOv8 推理 │
                                     └────────┬─────────┘
                          ┌───────────────────┼───────────────────┐
                  ┌───────▼──────┐    ┌────────▼────────┐  ┌────────▼───────┐
                  │  MySQL 数据库 │    │ 文件存储(uploads)│  │ 模型文件(weights)│
                  └──────────────┘    └─────────────────┘  └────────────────┘
```
- 前端层：渲染、交互、可视化（Canvas 缺陷框、ECharts），Axios 调后端 REST API。
- 后端 API 层：鉴权(JWT)、业务逻辑、任务编排、数据库读写、文件管理；不直接推理，经内部 HTTP 调算法服务（解耦）。
- 算法服务层：独立进程，加载模型，提供 `/infer`，输入图片返回结构化结果；单独部署便于替换模型。
- 数据层：MySQL 存业务数据，本地文件存图片与标注图，weights 存模型。
- 分层依据：前端管展示、后端管业务、算法管推理、数据库管存储，职责单一，便于 4 人并行与画包图/软件结构图。

交互原则：核心检测 ≤3 步；上传带拖拽与进度；结果区缺陷框 Canvas 醒目红框+标签。

## 17. 关键业务流程
**主流程（图片上传 → 检测结果输出）**
```
登录 → 进入检测页 → 选择/拖拽/拍照上传图片
  → 后端校验并存储图片，创建 product_sample
  → 创建 detection_task(processing)
  → 后端调用算法服务 POST /infer
       预处理 → 果实分割 → YOLOv8缺陷检测 → 特征计算 → 规则分级
  → 算法返回结构化结果+标注图
  → 后端写 detection_result + defect_label，关联参考价/溯源
  → task.status=done
  → 前端获取结果 → 渲染标注图+等级徽章+指标+缺陷列表
  → 用户可登记溯源/导出 PDF 报告
```
**辅助流程（管理员维护数据/模型）**
```
管理员登录 → 标准管理(调果径/着色率/缺陷阈值, is_active)
           → 价格管理(更新各等级参考价)
           → 模型管理(上传 best.pt → 登记 model_info → 激活 → 后续任务用新模型)
           → 样本/日志管理
所有写操作 → 记入 operation_log
```

## 20. 风险识别与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 数据不足/标注慢 | 训练无效果 | W1 启动采集；公开集打底；Roboflow 加速；自建集只做测试/演示；不贪规模 |
| 模型效果不稳定 | 漏检/误检 | **传统 CV 分级兜底**；调高置信度阈值保准确率；固定纯色背景实拍降难度；mAP 目标设演示级 0.6 |
| 无 GPU/训练慢 | 进度拖延 | Colab/AI Studio 免费 GPU；YOLOv8n 轻量；CPU 推理演示 |
| 周期紧 | 功能做不完 | 严守 MVP(P0)优先；P1 有余力做；P2 可砍；接口契约先行并行 |
| 前后端/算法联调难 | 集成卡壳 | §15 契约先定；算法服务先返 mock；Postman 逐接口验证 |
| 演示现场故障 | 答辩失分 | 本地离线演示+预录视频双保险；备已验证演示图集；脚本彩排 ≥2 次 |
| 文档不达标/抄袭 | 触发降档(红线) | 严格按模板排版；图原创；代码注释；引用标注 |
| 成员能力与假设不符 | 分工失衡 | W1 内组内自评特长，按 §18 对调 |

## 22. 验收标准（"可交付完成"定义）
1. 系统可运行：一键/文档化启动，前端可访问，登录正常。
2. MVP 闭环通过：上传(单/批量)→预处理→缺陷检测(框选)→分级(等级+指标)→可视化→入库→统计，无致命 bug。
3. 算法达演示级：缺陷 mAP@0.5 ≥0.6 或清晰可演示；分级人工一致率 ≥80%。
4. 加分功能可用：溯源+二维码、价格参考、PDF/Excel 导出至少全部可演示。
5. 文档齐全规范：需求(用例图+DFD1层+数据字典)、概要(软件结构图+E-R)、详细(类图+时序+活动+状态)、测试报告、用户手册、源代码文档，**排版符合要求**。
6. 交付物完整：报告(含任务书)+视频(≤50M)+源码包(含注释)+源代码文档，按 `第12组-《题目名》` 命名打包，**按时提交**。
7. 代码规范：业务逻辑有注释，结构清晰，可复现运行。
