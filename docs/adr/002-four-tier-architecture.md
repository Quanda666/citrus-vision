# ADR-002：采用四层 B/S 架构与算法服务解耦

- **状态（Status）**：已接受（Accepted）
- **日期**：2026-06-22
- **决策者**：第 12 组（组长 陈绍杰）

## 背景（Context）

系统需将 Web 业务（鉴权、上传、任务、报表）与计算密集的机器视觉推理结合。若把推理直接写进后端 API 进程，会带来：

- 推理阻塞 Web 请求，影响并发与响应。
- 模型替换需改动并重启整个后端。
- 算力无法独立扩展（如迁移到 GPU 机器）。
- 算法故障会拖垮整个后端服务。

## 决策（Decision）

采用**四层 B/S 架构**：前端（Vue3）/ 后端 API（FastAPI）/ 算法服务（独立 FastAPI 进程）/ 数据库（MySQL）+ 文件存储。

关键点：**算法服务独立成进程**，后端经内部 HTTP 接口 `POST /infer` 调用，二者解耦。

```
前端 ──HTTP/JSON──▶ 后端 API ──内部 HTTP /infer──▶ 算法服务
                       │
                   MySQL + 文件存储 + 模型权重
```

## 后果（Consequences）

**正面**：
- **模型可替换**：换 weights 或重启算法服务即可，不动后端。
- **算力可隔离**：算法服务可单独部署到 GPU 机器。
- **故障可降级**：算法服务异常时，后端走传统 CV 兜底，业务不全断（见 [ADR-001]）。
- **职责清晰**：后端管业务与鉴权，算法服务只做纯计算。
- 契约先行（`/infer` 接口）使前后端 + 算法可并行开发。

**负面 / 代价**：
- 多一次内部 HTTP 调用，有网络开销（局域网 / 本机可忽略）。
- 部署多一个服务，docker-compose 编排复杂度上升。

**关联**：部署拓扑见概要设计 §6；降级流程见详细设计 §7。
